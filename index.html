Hello, let's build things.


but how? Lets see...

...with building great things, comes great responsibilty.

What responsibilities?

A team of engineers is responsible for all their systems, which need to be able to scale to demand, be secure, be adaptable to changing business needs, be reliable, cost efficient and alert when issues arise.

Cloud Native Infrastructure

Running your services in the cloud allows you to be scale with demand, reduce server cost due to economies of scale, have increased security and use managed services your engineers don't have to maintain.

Deployment Pipelines

Developers change code often, and in unexpected ways! We need to deliver those changes in an automated way using deployment pipelines to ensure the same quality checks are done in every release.

System Monitoring

You've got code running in production, new changes come through the deployment pipeline... did it work? did it break things? Lets monitor so we know when changes impact users or the system.

System Alerting

At 3am the the unexpected happens, while all of your developers are asleep one of your services dies! No one is awake to check monitoring, so we alert and escalate that alert to a phone for on-call to deal with.

If you're building a big thing, then this responsiblity grows into building teams of engineers that work collaboratively together, are empowered to deliver as a individual teams, operate using agile delivery and upskill each other to become cross-functional individual contributors.

Cross Team Collaboration

Breaking a idea down into multiple teams to deliver the overarching goal is hard, these teams need to collaborate to deliver the idea without overlapping on work, or hindering each others progress with dependencies.

Empowered Teams

Making individual teams responsible for the tools they want to use, how they use them and how they maintain them empowers them to craft their own journey to delivering what they are accountable for.

Agile delivery

Things change, and much faster and more often than you'd think. Teams should be planning in iterative cycles to deliver continious value and have the ability to pivot their direction if the ship changes course.

Cross Functional Teams

Every individual of your team should be T shaped - have base understanding of a lot of things, and be an expert in few areas. Your team shouldn't say "Oh we can't release that to the database Jim's on holiday. Use pair programming to disperse knowledge across the team.

Constantly working towards balancing the satifaction of all of those will allow you to launch your 'thing'. Hard work eh?
